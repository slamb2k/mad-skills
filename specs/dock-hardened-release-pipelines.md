# /dock — Hardened Release Pipelines

**Status:** Draft
**Date:** 2026-07-06
**Owner:** slamb2k
**Skill:** `skills/dock`

## Summary

Port the hardening patterns that protect mad-skills' own release workflow
(`.github/workflows/ci.yml`) into the container release pipelines that `/dock`
generates for other projects. The npm-specific mechanisms translate to their
OCI-container equivalents:

| mad-skills (npm release) | `/dock` container equivalent |
|--------------------------|------------------------------|
| Trusted publishing (npm OIDC, no `NODE_AUTH_TOKEN`) | GitHub OIDC → federated cloud-registry auth (ECR/ACR/GAR); GHCR keeps its ephemeral `GITHUB_TOKEN` |
| `npm publish --provenance` | Keyless **cosign** signing + SLSA build provenance/attestation |
| `npm view <pkg>@<v>` publish-guard (skip if already published) | `crane`/`docker manifest inspect` existence guard before build/push/retag |
| Publish-before-git-write, idempotent recovery mode | Promote by immutable `@sha256:` digest; re-runnable retag/promote (recovery mode) |
| `concurrency:` group, autostash rebase, tag guards | `concurrency:` guards + hardened promote/tag steps |

The generated pipelines are **secure by default** (hardened is the only mode),
matching mad-skills' own always-on posture. Because OIDC and keyless signing
require one-time setup on the user's cloud side, `/dock` also generates a
`deploy/SETUP.md` with the exact steps and surfaces a **🔐 Required cloud setup**
section in its final report.

## Goals

- Generated GitHub Actions and Azure Pipelines use **keyless OIDC** to
  authenticate to cloud registries — no long-lived cloud credentials stored in
  CI secrets.
- Pushed images are **signed and carry provenance**; promotion **verifies** the
  signature before deploying.
- Build/push/retag steps are **idempotent** — safe to re-run without duplicate
  work or collisions (recovery mode).
- Environments are promoted by **immutable digest**, never a mutable tag — the
  strongest form of build-once-promote-everywhere.
- The user is told **exactly** what cloud-side trust to configure for first run.

## Non-Goals

- **GitLab CI** hardening — GitLab stays on its current credential-based
  template. GitLab is not otherwise a first-class mad-skills platform.
- **Private Rekor / air-gapped sigstore** infrastructure — keyless cosign
  (public Fulcio/Rekor) is the default; a key-pair fallback is documented but
  not generated.
- Changing dock's **non-container** behavior or any other skill.
- Retrofitting pipelines already generated by a prior `/dock` run beyond dock's
  existing idempotency rules (ask-before-overwrite).

## Approach

**Inline, secure-by-default** (approach A from brainstorming). The GitHub Actions
and Azure Pipelines sections of `references/pipeline-templates.md` are rewritten
so the hardened variant is the *only* variant. No dual "hardened / unhardened"
templates — since hardened is always-on there is no non-hardened path worth
maintaining, and dual templates would drift. GitLab's section is left untouched.

Rejected alternatives:
- **B — separate `hardening.md` layer that Phase 3 composes onto base templates.**
  Dock templates are copy-paste starting points; layering YAML fragments at
  generation time is awkward and error-prone.
- **C — parallel `*-hardened` template variants.** Most files, highest drift
  risk, and pointless when hardened is the only mode.

### Pattern 1 — OIDC keyless registry auth

Generated GitHub Actions workflow:
- Top-level (or job-level) `permissions:` includes `id-token: write` and
  `contents: read` (plus `packages: write` for GHCR).
- **GHCR** (`PLATFORM == github` default): keep the ephemeral
  `docker/login-action` + `secrets.GITHUB_TOKEN` — already short-lived and
  OIDC-backed; no change needed.
- **Cloud registries** emit an OIDC login step with **no stored keys**:
  - **AWS ECR** → `aws-actions/configure-aws-credentials@v4` with
    `role-to-assume` (no access-key secrets) → `aws ecr get-login-password`.
  - **Azure ACR** → `azure/login@v2` with Workload Identity Federation
    (`client-id`/`tenant-id`/`subscription-id`, no client secret) →
    `az acr login`.
  - **Google Artifact Registry** → `google-github-actions/auth@v2` with Workload
    Identity Provider (`workload_identity_provider` + `service_account`, no SA
    key JSON) → `gcloud auth configure-docker`.

Azure Pipelines: use a **Workload Identity Federation service connection**
(`azureSubscription:`) for ACR login instead of a stored registry password /
service-principal secret.

### Pattern 2 — Signing + provenance

- Build with `docker/build-push-action` using `provenance: true` and
  `sbom: true` (buildx SLSA provenance + SBOM attestations).
- After push, **keyless** `cosign sign` against the image **digest** (Fulcio
  cert + Rekor transparency log; `COSIGN_EXPERIMENTAL`/`--yes`, backed by the
  `id-token: write` OIDC token — no signing key in secrets).
- The **promote** job runs `cosign verify` (checking the certificate identity =
  the workflow's OIDC issuer + `repo:OWNER/REPO`) **before** deploying to any
  environment. Verification failure blocks promotion.
- Azure Pipelines: same cosign steps driven by the WIF service connection's
  federated token.

### Pattern 3 — Idempotent push/retag guards

- Before build, a guard step runs `crane manifest <ref>` (or
  `docker manifest inspect`) on the target tag/digest. If the artifact already
  exists, **skip build+push** and reuse the existing digest — the container
  analog of the `npm view <pkg>@<v>` guard.
- The promote/retag job is **re-runnable**: retag/attach steps tolerate an
  already-promoted digest (check-then-act) and exit cleanly ("recovery mode")
  rather than failing a re-run, mirroring the release job's tag-exists recovery
  branch.
- `concurrency:` group keyed on workflow + ref with `cancel-in-progress` (GH),
  and the Azure equivalent, to prevent overlapping release runs from racing.

### Pattern 4 — Digest-pinned promotion

- The build job outputs the pushed image reference as `IMAGE@sha256:<digest>`
  (a job output / step output consumed downstream).
- Dev, staging, and prod deploy steps reference the **digest**, never a mutable
  tag. Human-facing tags (`latest`, `vX.Y.Z`) may still be *attached* for
  discoverability, but deployment always resolves the digest.
- `deploy/environments.yml` `registry_tag_pattern` is replaced/augmented with a
  resolved digest reference so the environment matrix records exactly what was
  deployed.

### Setup callouts (the secure-by-default gap)

Phase 3 generates `deploy/SETUP.md` tailored to the chosen registry/cloud,
containing copy-pasteable steps:
- Create the **federated credential / IAM role trust** scoped to
  `repo:OWNER/REPO:ref:refs/heads/main` (and tag refs for release), for the
  chosen cloud (AWS IAM role + trust policy / Azure app federated credential /
  GCP Workload Identity Pool + provider).
- Grant that identity **registry push** permission.
- Note that keyless cosign writes to the **public Rekor** transparency log;
  include a one-line **key-pair fallback** (`cosign generate-key-pair`, key in
  CI secret) for orgs that cannot use public transparency infra.

The Final Report's **🔐 Required cloud setup** section lists the 2–4 concrete
actions and points at `deploy/SETUP.md`.

## Testing

- `skills/dock/tests/evals.json` assertions on a representative generation:
  - GH workflow contains `id-token: write`.
  - GH workflow contains a `cosign sign` and a `cosign verify` (verify in the
    promote job).
  - Build step sets `provenance: true`.
  - Promote step references a `@sha256:` digest, not a bare tag.
  - `deploy/SETUP.md` is listed among generated files.
- `npm run validate && npm run lint` pass for the modified skill.
- Manual: run `/dock --dry-run` against a sample Node app and eyeball the
  generated GH + Azure workflows and `SETUP.md`.

## Open Questions / Assumptions

- **Assumption:** keyless cosign (public Fulcio/Rekor) is an acceptable default;
  the private-Rekor case is documented, not generated. *(Confirmed in
  brainstorming.)*
- **Assumption:** `crane` is an acceptable guard tool to reference in generated
  pipelines (installed via `imjasonh/setup-crane` on GH). If undesirable, fall
  back to `docker manifest inspect` only.
- **Deferred:** GitLab CI hardening — tracked as a possible follow-up spec, not
  in this scope.
